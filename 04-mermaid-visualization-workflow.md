## 結論

この記事の本質は、**Mermaidを「構造固定用の中間言語」として使い、Gemini画像生成に清書だけを任せると、エンジニア向け図をビジネス向け図解へ低コスト変換できる**という話です。

単なる「AIで図を作る」ではなく、
**構造設計＝人間・Mermaid**
**見た目の清書＝Nano Banana / Gemini画像生成**
に役割分担するのがポイントです。

---

## 理由

長文説明をそのまま画像生成AIへ投げると、AIが勝手に要約してしまい、要素抜け・矢印の向きの崩れ・関係性の破綻が起きやすい。

そこで先にMermaidで、

* 登場要素
* 関係
* 矢印
* 流れ
* 階層

を固定してから画像生成に渡す。

これにより、AI側の仕事を「考える」ではなく「清書する」に限定できるため、資料向けの安定した図解になりやすい、という主張です。

---

## 記事の要点

使い方はかなり単純です。

1. 普段通りMermaidで構成図・フロー・シーケンス図を書く
2. 専用GemにMermaidを貼る
3. 白背景・日本語ゴシック体・アイコン付きの図解画像として出力する

Gemのカスタム指示も短く、内容としては、

* 入力を要約・構造化する
* 日本語ゴシック体にする
* 公式アイコンを使う
* 長文なら要約してから構造化する
* 背景白、タイトル不要

というものです。

記事では、AWS 3層Webアプリ構成図、データ分析基盤パイプライン、業務フローの3例を挙げています。

---

## 有力シナリオ

### 1. 技術者向けREADME・設計書の清書にかなり使える

これは黒澤さんの個人開発文脈だと、かなり相性が良いです。

たとえば、

* private-cloud-db のDB役割分担図
* Turso / Neon / local PostgreSQL の同期構成
* ニュースバッチのCloud Run / local Mac / Discord / Retool連携図
* お気に入りDBの共通資産化フロー
* Production Readiness Review の運用フロー

こういうものは、まずMermaidで構造を固定し、その後に図解化すると「自分用メモ」から「人に見せられる資料」に変換しやすいです。

### 2. draw.ioやFigmaの代替ではなく、初稿生成ツールとして強い

厳密な図面ツールではありません。

ただし、初稿としては強いです。

特に、AIにコードやREADMEからMermaidを作らせて、それをさらに画像化する流れにすると、

**コード/設計書 → Mermaid → 図解画像**

という自動清書パイプラインが作れます。

これは「毎回ゼロから図を作る」より明らかに優位性があります。

### 3. ビジネス側説明用には効果が高い

Mermaidはエンジニアには読みやすいですが、非エンジニアには少し硬い。

この記事のワークフローは、
**エンジニア内部の正確性**と
**ビジネス側への伝達性**
の間を埋める手法です。

## 破綻条件

この手法が崩れる条件は明確です。

### 1. 公式アイコンの厳密性が必要な場合

「公式アイコンを使って」と指示しても、完全なAWS公式アイコンになるとは限らない。

社外向け資料、ブランドガイドライン準拠資料、正式提案書では、そのまま使うのは危険です。
この場合は、AI出力を下書きにして、最後は手で差し替える前提が必要です。

### 2. 日本語文字の品質が弱いモデルを使う場合

記事では、初代Nano Banana相当だと日本語が崩れやすいとされています。

日本語テキスト入り図解では、モデル選択が重要で、記事内では **Pro または 2** を推奨しています。

### 3. Mermaidが大きすぎる場合

ノードが20〜30個あるような大規模図を一発で投げると、要素が間引かれる可能性があります。

この場合は、

* サブグラフ単位
* レイヤー単位
* 業務フロー単位
* DB同期単位
* 通知単位

に分割するほうが安定します。

### 4. 社外秘情報をそのまま入れる場合

構成図には、ホスト名、IP、DB名、認証情報、内部サービス名が入りがちです。

クラウドAIに投げるなら、ダミー化は必須です。

## 実務・行動への影響

黒澤さんの用途では、これはかなり使えます。

特におすすめは、**「Claude Code / ChatGPTにMermaidを書かせる → Gemini系で図解化」**の流れです。

実務的には、次のように使うのがよさそうです。

```text
README / 設計書 / ADR
        ↓
AIにMermaid化させる
        ↓
Mermaidを人間が軽く修正
        ↓
Gemini画像生成で白背景ビジネス図解化
        ↓
README / Qiita / Notion / Slack / Discord / PR説明に貼る
```

特に価値が出るのは、以下です。

* 個人アプリ群の全体構成図
* DB役割分担図
* ローカル処理とクラウド処理の分離図
* ニュース取得バッチの流れ
* Retool / Neon / Turso / PostgreSQL の責務分離
* Production Readiness Review のチェックフロー
* AI開発ワークフローの説明図

この手法の価値は、**プロンプトの巧さより、Mermaidを挟むことで構造責任をAIから奪う点**です。

つまり、AIに丸投げして破綻するのではなく、
**人間が構造を握り、AIには見栄えだけを外注する**。

この分担はかなり正しいです。
