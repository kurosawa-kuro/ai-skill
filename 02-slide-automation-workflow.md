思考時間: 9s
結論

この記事の本質は、「AIにスライドを作らせる」ではなく、「AIが迷わない制作ラインを作る」話です。

特に重要なのは、Marp + Claude Code + Skills を使って、以下を仕組み化している点です。

スタイルガイドを作る
レイアウトパターンを部品化する
Claude Code Skill に作業手順を埋め込む
Playwright系のブラウザ確認で見た目崩れを検出する
画像生成・SVG生成・過去スライド検索・レビュー導線まで統合する

つまり、単発生成ではなく、**「スライド制作の小さな開発基盤」**を作っている記事です。

理由

最初の問題意識は、Claude Codeに普通にスライド作成を頼むと、毎回デザイン・余白・色・フォント・レイアウトがバラつくことです。そこで著者は、過去スライドを分析して docs/style-guide.md にデザイン哲学・色・フォントサイズ・レイアウトルールを落とし込みます。

次に、毎回ゼロからHTML/CSSを書くのを避けるため、slides/example.md に 7カテゴリ・40種類のレイアウトパターンを用意しています。タイトル、セクション、2カラム比較、3カラム、グリッド、パネル、画像背景、統計表示、QRコードなどをテンプレート化する発想です。

さらに、それらを Claude Code Skill 化しています。たとえば slide-style-rector は、対象スライドを読み、slides/example.md から適切なパターンを選び、スタイルガイドに沿って整形する役割です。

かなり重要なのが layout-fix です。Tailwind + Marp はコード上は正しくても、実レンダリングでは下が切れる、重なる、詰まることがある。そのため Playwright系の agent-browser でスクリーンショットを撮り、Claude Codeに視覚確認させ、gap・margin・padding・カラム数などを修正させる流れを作っています。

有力シナリオ

黒澤さんの個人アプリ開発に引き寄せると、これはスライド専用ノウハウではなく、AIコーディング用の「作業OS化」ノウハウとして見るのが有力です。

一番使える構造はこれです。

docs/style-guide.md
  → 判断基準・見た目・文体・設計思想

examples/
  → 正解パターン集・テンプレート集

.claude/skills/
  → Claude Codeに読ませる作業手順

scripts/
  → 検証・スクショ・lint・build・cleanup

feedback/
  → 人間のレビュー導線

これは、今やっている個人アプリ群にもそのまま応用できます。

たとえば Retool / PWA / ニュースアプリ / お気に入りDB / リネーマーなら、以下のように置き換えられます。

docs/app-style-guide.md
  → UI原則、カード表示、一覧表示、検索導線、文体

docs/db-boundary-guide.md
  → Turso / Neon / Local PostgreSQL の責務分離

examples/query-patterns.md
  → よく使うSELECT、WHERE、同期パターン

examples/ui-patterns.md
  → カード、一覧、詳細、リンク集、ランチャー画面

.claude/skills/refactor-boundary.md
  → 散乱機能の統合、防腐層、責務分離

.claude/skills/retool-ui-review.md
  → 表示崩れ、カード密度、モバイル導線確認

この記事の価値は、AIに自由作文させるほど品質が落ちるので、選択肢を絞った上で高速に働かせるという点です。

破綻条件

この方式が破綻するのは、次の条件です。

第一に、スタイルガイドやパターン集が弱い場合。AIに渡す「正解例」が少ないと、結局また毎回違うものを生成します。

第二に、パターンが増えすぎて管理不能になる場合。40パターンは強いですが、メンテしないと「古い正解例」がAIに混ざります。

第三に、見た目検証を人間の目だけに戻す場合。この記事の肝は、コード生成後にスクショで視覚検証する点です。ここを省くと、MarpでもWeb UIでも「コードは合ってるが画面は崩れている」が残ります。

第四に、Skillが単なるプロンプト集になる場合。slide-style-rector、layout-fix、svg-creator、marp-lens のように、入力・参照ファイル・検証手順・出力先まで決めているから機能しています。

実務・行動への影響

黒澤さんの場合、この記事から取るべき実務アクションは 「Claude Code Skillをスライド用ではなく、個人アプリ開発の反復作業に転用する」 です。

優先度順ならこれです。

docs/00_ai_workflow.md を作る
AIに何を読ませ、何を正本にし、どの順で作業するかを固定する。
docs/patterns/ を作る
DB設計、UIカード、一覧、同期、リファクタ、テスト追加の正解パターンを保存する。
.claude/skills/refactor-cleanup/SKILL.md を作る
「似た機能の散乱」「共有化」「単一責任」「防腐層」「後方互換レガシー整理」「不足テスト追加」を、黒澤さん用の標準Skillにする。
スクショ検証を入れる
PWAやRetool代替UIを作るなら、Playwrightでスクショを取り、AIに画面崩れを直させるループを作る。
過去成果物検索を入れる
marp-lens 的に、過去README、設計書、仕様書、実装例を検索できる導線を作る。これは個人アプリ群が増えるほど効きます。

この記事は、単なるMarp記事ではなく、AI時代の「自分専用開発標準ライブラリ」を作る話として読むのが一番リターンが高いです。
